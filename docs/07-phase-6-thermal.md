# Phase 6 (опционально): Термальный мониторинг NVAPI/ADL + расширенная аналитика служб

## Цель фазы

- **GPU:** температура, троттлинг-статус, клоки, power limit через официальные SDK NVIDIA (NVAPI) и AMD (ADLX) — без kernel-драйверов;
- **CPU:** честный мониторинг того, что доступно из user-mode (частоты, power limits, thermal-события ОС), с открытым признанием ограничений (точная T° ядер без драйвера недоступна на большинстве систем);
- **Троттлинг-детекция:** факт и длительность троттлинга по данным ОС/вендоров, корреляция с трендами (Phase 2): «после чистки от пыли троттлинг исчез» — видимо на графике;
- **Аналитика служб:** ресурсные профили служб во времени (CPU/IO per service) — какие фоновые службы реально стоят ресурсов.

Измеримо: GPU-температуры совпадают с HWiNFO/vendor-утилитами ±2°C; троттлинг-события детектируются в синтетическом стресс-тесте; per-service профили строятся за 24 ч фона.

## Границы (Scope)

**Входит:** NVAPI (NVIDIA), ADLX (AMD GPU), CPU user-mode метрики (PDH/PowerCfg/WMI), троттлинг-детекция, интеграция в тренды, per-service ресурсная аналитика.

**НЕ входит:**
- Kernel-mode драйвер для чтения MSR (точные CPU-температуры) — **зафиксированный отказ** v1: подпись, WHQL, HVCI-совместимость и риск BSOD перевешивают ценность. Пересмотр — отдельным решением после 1.0.
- Разгон/андервольт/управление кулерами — только мониторинг.
- Intel GPU: базово через ту же D3DKMT-статистику; IGCL SDK — backlog.

## Технические задачи

### 6.1 NVIDIA — NVAPI

- SDK: NVAPI (лицензия допускает redistribution в приложениях — перепроверить актуальную редакцию). Динамическая загрузка `nvapi64.dll` (`nvapi_QueryInterface`), graceful-отсутствие при не-NVIDIA системе.
- Данные: `NvAPI_GPU_GetThermalSettings` (температуры GPU/памяти/hotspot где доступно), `NvAPI_GPU_GetAllClockFrequencies`, `NvAPI_GPU_GetDynamicPstatesInfoEx` (загрузка), perf-limit причины (`NvAPI_GPU_PerfPoliciesGetStatus` — thermal/power/voltage limit флаги: это и есть прямой троттлинг-сигнал).
- Обёртка — collector `thermal.nvidia`, polling 1–2 с при открытом экране, 60 с фоном.

### 6.2 AMD — ADLX (не legacy ADL)

- ADLX — текущий SDK AMD (ADL — deprecated; использовать ADLX с fallback на ADL для старых Adrenalin). Динамическая загрузка `amdadlx64.dll`.
- `IADLXGPUMetrics`: temperature, hotspot, clocks, power, fan; `IADLXPerformanceMonitoringServices` для polling. Троттлинг — по метрикам power/thermal limit.
- Риск: ADLX требует установленного Adrenalin; при его отсутствии — «данные недоступны, установите драйвер AMD» (честная деградация).

### 6.3 CPU user-mode (без драйвера)

Доступно легально из user-mode:

- Частоты/лимиты: PDH `\Processor Information(*)\% Processor Performance`, `Processor Frequency` — фактическая частота против базовой = косвенный троттлинг-признак; `CallNtPowerInformation(ProcessorInformation)` — текущий/максимальный Mhz per core.
- Power/thermal события ОС: `Microsoft-Windows-Kernel-Power` ETW/event log (thermal zone events), WMI `MSAcpi_ThermalZoneTemperature` (работает далеко не везде и часто отдаёт температуру зоны, не ядер — показывать с пометкой «ACPI thermal zone, не температура ядер», только если валидно меняется под нагрузкой — иначе скрыть);
- PPM: `PowerGetActiveScheme`/powercfg-данные — power throttling процессов (`PROCESS_POWER_THROTTLING_STATE` через `GetProcessInformation`).
- **UI обязан прямо говорить:** «Точная температура ядер CPU требует kernel-драйвера, который мы сознательно не ставим. Показываем частотный троттлинг и данные ACPI» — это фирменная честность продукта.

### 6.4 Троттлинг-детекция и тренды

- Детектор: окно N секунд, признаки — GPU perf-limit флаги (thermal), CPU `% Processor Performance` < 100−δ при высокой загрузке, Kernel-Power thermal события. Каждое срабатывание — событие в SQLite (`thermal_events`), связывается с трендами Phase 2.
- Экран: текущие значения + история троттлинг-эпизодов (когда, длительность, вероятная причина: thermal vs power limit).

### 6.5 Аналитика служб

- Источник: цикл per-process сэмплинга Phase 1 (`NtQuerySystemInformation`) + маппинг PID→службы (`I_QueryTagInformation` или `EnumServicesStatusExW` с PID; для shared `svchost` — разбиение по service tag: `ScQueryServiceTagInfo`-путь недокументирован, использовать документированную декомпозицию: на Win10+ большинство служб уже в отдельных процессах при RAM > 3.5 GB).
- Агрегация: CPU-время и IO per service в минутные ряды (та же trend-инфраструктура), топ-потребители за 24 ч/7 дней.
- Связка с UI служб Phase 1: «эта служба потребила X CPU-минут за неделю» — данные для решения об отключении вместо догадок.

## Зависимости от предыдущих фаз

- Phase 0: контракт collectors, storage.
- Phase 1: per-process сэмплер, экран служб.
- Phase 2: trend-инфраструктура, фоновые снапшоты.
- Phase 5: explainability-слой (все термальные метрики обязаны иметь качественные объяснения ограничений).

## Риски и митигация

| Риск | Митигация |
|------|-----------|
| Лицензии NVAPI/ADLX ограничивают распространение | Юр-проверка **до старта фазы**; SDK линкуются динамически, при запрете — деградация модуля |
| Вендорские DLL падают/висят (драйверные баги) | Все вызовы SDK — в отдельном sandbox-процессе (`thermal-agent.exe`); его крэш не трогает приложение; watchdog + перезапуск |
| MSAcpi даёт мусор на половине машин | Валидационный фильтр (значение реагирует на нагрузку, в физичных пределах), иначе скрыть; никогда не показывать сомнительные цифры |
| Ожидания пользователей «где температура CPU как в HWiNFO» | Явное объяснение в UI + FAQ; позиционирование «без kernel-драйвера — сознательно» |
| Polling жрёт батарею ноутбука | Адаптивная частота (экран закрыт → 60 с), пауза на батарее по умолчанию (настройка) |
| svchost-shared службы неточно атрибуцируются | Честная пометка «общий процесс, атрибуция приблизительная»; не выводить точные цифры там, где их нет |

## Критерии готовности (Definition of Done)

- [ ] NVIDIA и AMD машины: температуры/клоки совпадают с HWiNFO ±2°C/±3%; отсутствие вендора → модуль чисто скрыт.
- [ ] Синтетический стресс (FurMark/prime95-класс, на тестовом стенде): троттлинг-эпизод детектируется, попадает в историю, причина классифицирована.
- [ ] Крэш thermal-agent (kill) не влияет на приложение; агент перезапускается.
- [ ] Per-service топ за 24 ч на dogfood-машинах согласуется с Process Explorer-наблюдениями.
- [ ] На ноутбуке от батареи фоновый мониторинг ≤ 0.5% среднего CPU, настройка паузы работает.
- [ ] Все метрики покрыты explainability, включая тексты об ограничениях CPU-температур.

## Тестирование

- **Unit:** детектор троттлинга (фикстуры временных рядов), валидатор ACPI-температур, агрегация per-service.
- **Интеграционные:** мок-слой vendor SDK (фейковые DLL-интерфейсы) для CI; реальные прогоны — на стендах NVIDIA/AMD.
- **Стресс:** длительный (24 ч) мониторинг под перемежающейся нагрузкой — утечки памяти агента, стабильность polling.
- **Ручное:** сравнение с HWiNFO/GPU-Z/vendor-утилитами на каждой конфигурации; ноутбук в Modern Standby (мониторинг обязан корректно засыпать/просыпаться).

## Оценка трудозатрат

**4–6 недель.** NVAPI ~1 нед; ADLX ~1–1.5 нед; sandbox-агент+watchdog ~0.5 нед; CPU-метрики+детектор ~1 нед; per-service аналитика ~1 нед; тесты на стендах ~1 нед (перекрывается).

## Открытые вопросы

- Покупка/поиск тестовых стендов (минимум: 1×NVIDIA, 1×AMD GPU, 1×ноутбук с троттлингом) — организационный блокер.
- Intel Arc/IGCL — добавлять ли в эту фазу при наличии времени или строго backlog.
- Пересмотр решения о kernel-драйвере после 1.0 (отдельный RFC с оценкой WHQL-процесса) — назначить срок.
