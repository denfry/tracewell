# Phase 3: Безопасная чистка диска (без реестра!) с dry-run и откатом

## Цель фазы

Модуль чистки, который находит и удаляет только то, что можно объяснить и вернуть:

- дубликаты файлов (по содержимому, не по имени);
- сироты инсталляторов (`%WINDIR%\Installer` orphaned patches, брошенные `$PatchCache$`, устаревшие пакеты в `SoftwareDistribution\Download`);
- старые restore points (кроме последнего) и shadow copies — с точным размером;
- кэши браузеров (Chrome/Edge/Firefox) — только cache, не куки/пароли/история.

Каждая категория: **dry-run по умолчанию**, объяснение «что это и почему безопасно», карантин с восстановлением, журнал. Никаких «найдено 47 проблем» — только байты и факты.

Измеримо: ни одного случая потери пользовательских данных на тестовой матрице; всё удалённое (кроме restore points/shadow copies — они невозвратны и помечены особо) восстанавливается из карантина в 1 клик в течение 14 дней.

## Границы (Scope)

**Входит:** 4 категории выше; карантин-инфраструктура; dry-run отчёты; расчёт реального высвобождаемого места (с учётом hardlink'ов и компрессии).

**НЕ входит:**
- Чистка реестра — **никогда** (зафиксировано философией продукта; в UI объяснить почему её нет).
- «Оптимизация RAM», префетч-чистка и прочие плацебо.
- Временные файлы `%TEMP%` в v1 (низкая ценность, высокая конфликтность с запущенным софтом) — перенесено в backlog.
- Чистка облачных папок (OneDrive/Dropbox placeholder'ы) — детектировать и исключать.

## Технические задачи

### 3.1 Карантин (фундамент фазы — делается первым)

- Расположение: `%SystemDrive%\Tracewell.Quarantine\<action_id>\` (тот же том — перемещение = `MoveFileExW` без копирования, мгновенно и атомарно в пределах тома; для других томов — карантин на том же томе, что и файл).
- Метаданные в SQLite: исходный путь, ACL (`GetNamedSecurityInfoW` → бэкап SDDL), атрибуты, таймстемпы, hash (BLAKE3), срок хранения (14 дней, настраиваемо).
- Восстановление: обратный `MoveFileExW` + восстановление ACL/атрибутов; конфликт имён — диалог.
- Автоочистка карантина по сроку — фоновой задачей, с записью в журнал.
- Файлы, занятые процессами: не трогаем (открытие с `FILE_SHARE_NONE`-пробой), в отчёте — «занят, пропущен».

### 3.2 Дубликаты

- Пайплайн: enum (`FindFirstFileExW` c `FIND_FIRST_EX_LARGE_FETCH` или `NtQueryDirectoryFileEx` для скорости) → группировка по размеру → частичный хэш (первые+последние 64 KB) → полный BLAKE3 только для оставшихся кандидатов. I/O — последовательный по группам, low-priority (`SetPriorityClass` + `SetFileBandwithReservation` нет, использовать `FileIoPriorityHintInfo` = Low).
- Обязательные исключения: hardlink-дубли (`GetFileInformationByHandle` → `nNumberOfLinks`/FileId — это один файл!), reparse points, OneDrive/Dropbox/GDrive placeholder'ы (`FILE_ATTRIBUTE_RECALL_ON_DATA_ACCESS`), `%WINDIR%`, `Program Files` (системные дубли — норма: WinSxS), профили других пользователей.
- Область поиска — только выбранные пользователем папки (по умолчанию — пользовательские библиотеки), не «весь диск» молча.
- UX: группы дублей, оригинал выбирается правилом (старейший / в лучшем пути) с ручным override; удаление → карантин.

### 3.3 Orphaned installer files

- `%WINDIR%\Installer`: сопоставление `.msi`/`.msp` с зарегистрированными продуктами через MSI API: `MsiEnumProductsExW` + `MsiGetProductInfoExW(INSTALLPROPERTY_LOCALPACKAGE)` и `MsiEnumPatchesExW` + `INSTALLPROPERTY_LOCALPACKAGE` для патчей. Файл, на который не ссылается ни один продукт/патч — кандидат-сирота.
- Консервативность: перед пометкой перечитать список **дважды с интервалом** (инсталляция могла идти в момент скана); msi/msp занятые — пропуск; **только карантин, срок 30 дней** (удаление сироты, который вдруг нужен для uninstall — болезненно).
- `SoftwareDistribution\Download`: только содержимое старше 30 дней и при остановленной службе wuauserv не трогаем — вместо ручного удаления предпочтительно вызвать штатную очистку (запуск `cleanmgr`-handler'а не нужен: достаточно удалить содержимое папки при остановленном wuauserv через elevated helper, с явным объяснением).

### 3.4 Restore points / Shadow copies

- Перечисление: COM VSS `IVssBackupComponents::Query` (`VSS_OBJECT_SNAPSHOT`) — ID, дата, том; размер занятого хранилища — `IOCTL_VOLSNAP`-путей избегать, использовать данные `vssadmin list shadowstorage`-эквивалента через `IVssDifferentialSoftwareSnapshotMgmt::QueryDiffAreasForVolume`.
- Удаление конкретной точки: `IVssBackupComponents::DeleteSnapshots(VSS_OBJECT_SNAPSHOT, id)`. Всегда сохраняем самую свежую. Помечено «НЕВОЗВРАТНО» красным — единственная категория без карантина; подтверждение с вводом или двойным кликом-холдом.
- Требует admin → elevated helper.

### 3.5 Кэши браузеров

- Пути: Chrome/Edge — `%LOCALAPPDATA%\{Google\Chrome,Microsoft\Edge}\User Data\<Profile>\Cache\Cache_Data`, `Code Cache`, `GPUCache`; Firefox — `%LOCALAPPDATA%\Mozilla\Firefox\Profiles\*\cache2`. Профили перечислять по `Local State` (JSON)/`profiles.ini`, не хардкодить Default.
- Только каталоги кэша. Явно НЕ трогаем: Cookies, Login Data, History, Sessions, Extensions.
- Запущенный браузер: детект по lock-файлам/процессам → предложить закрыть или пропустить (не убивать процессы).
- Кэш не карантинится (регенерируемый по определению, объём большой) — помечено в UI «восстановится сам при сёрфинге».

### 3.6 Dry-run и отчёт

- Каждый cleaner реализует `ICleaner { scan() -> CleanupPlan; execute(plan, mode) }`, `mode ∈ {DryRun, Quarantine, Delete}`; DryRun — default. `CleanupPlan` сериализуется в снапшот (история сканов — тоже тренд).
- Отчёт: по каждой позиции — путь, размер, причина-кандидатуры, метод проверки, обратимость (карантин N дней / невозвратно / самовосстановится).

## Зависимости от предыдущих фаз

- Phase 0: storage, снапшоты, журнал ошибок.
- Phase 1: ActionJournal + elevated helper (VSS, wuauserv, Installer), UI-каркас, стоп-лист-механика.

## Риски и митигация

| Риск | Митигация |
|------|-----------|
| Удаление нужных данных (главный риск продукта) | Карантин-по-умолчанию, консервативные детекторы, dry-run default, стоп-листы путей, двойной скан для installer; beta-период фазы длиннее обычного |
| Hardlink/junction/симлинки → «освобождение» пустоты или удаление оригинала | FileId-дедупликация, пропуск reparse points всегда; unit-тесты на все типы линков |
| Облачные placeholder'ы: хэширование триггерит скачивание гигабайт | Фильтр `FILE_ATTRIBUTE_RECALL_ON_DATA_ACCESS`/`OFFLINE` до открытия файла; тест с OneDrive «files on demand» |
| Удаление restore point'а перед сбоем пользователя | Всегда сохраняем последний; UX-фрикция на подтверждении; показ даты и что «до» этой точки откатиться станет нельзя |
| Антивирусы реагируют на массовые файловые операции | Троттлинг операций, отсутствие shell-инжекций, обычные API; уведомить Defender-portal о поведении |
| Долгий скан больших дисков блокирует UX | Стриминговый прогресс, отмена в любой момент (CancellationToken из Phase 0), сохранение частичных результатов |

## Критерии готовности (Definition of Done)

- [ ] На тестовой матрице (включая OneDrive-профиль, hardlink-фикстуры, запущенные браузеры) — ноль потерь данных; негативные тесты доказывают, что защитные фильтры срабатывают.
- [ ] Восстановление из карантина возвращает файл с исходными ACL/атрибутами/таймстемпами (побайтовое сравнение + SDDL).
- [ ] Dry-run отчёт точен: предсказанные байты совпадают с фактически освобождёнными в пределах 5%.
- [ ] Kill процесса посреди execute → журнал позволяет корректно завершить/откатить операцию при следующем запуске (crash-recovery тест).
- [ ] Скан дублей 500 GB данных — фоновым приоритетом, UI отзывчив, отмена < 1 сек.
- [ ] Формулировки всех объяснений отревьюены на «отсутствие запугивания».

## Тестирование

- **Unit:** пайплайн дедупликации (фикстуры), детектор сирот MSI (мок MSI API), планы/сериализация, карантин-метаданные.
- **Интеграционные (только VM с чекпоинтами):** полный цикл scan→dry-run→quarantine→restore на подготовленных дисковых образах: hardlink'и, junctions, cloud placeholders, кириллические/длинные (>260, `\\?\`) пути, FAT32-флешка.
- **Crash-тесты:** TerminateProcess на каждом этапе execute; power-cut VM.
- **Ручное:** реальные захламлённые машины (dogfood), сравнение находок с WizTree/Everything-верификацией.
