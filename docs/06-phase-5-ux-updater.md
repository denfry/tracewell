# Phase 5: Полировка UX, explainability-слой, автообновление — релиз 1.0

## Цель фазы

Продукт готов к публичному релизу 1.0:

- **Explainability-слой** как системная фича: каждое значение, статус и рекомендация имеют структурированное объяснение (что, как измерено, источник, погрешность, что будет при действии, как откатить);
- UX-полировка: onboarding, пустые состояния, ошибки, доступность, производительность UI;
- **Автообновление приложения**: подписанное, с дельтами не обязательно, но с проверкой подписи и постепенным (staged) раскатом;
- Дистрибуция: подпись кода, SmartScreen-репутация, инсталлятор публичного качества.

Измеримо: новый пользователь без инструкции проходит сценарии «понять автозагрузку», «проверить диск», «почистить безопасно» (успех ≥ 80% в коридорных тестах); апдейт со старой версии на новую — автоматический, с сохранением БД и настроек.

## Границы (Scope)

**Входит:** explainability-инфраструктура и контент; onboarding/empty/error-состояния; доступность (Narrator, клавиатура, контраст); локализация-инфраструктура (en + ru); автообновление; подпись кода; телеметрия крэшей (opt-in, локальные дампы + добровольная отправка); лицензия/EULA/privacy policy; сайт-страница загрузки.

**НЕ входит:** новые диагностические модули; облачные аккаунты; A/B-механики; магазин (Microsoft Store — рассматривается после 1.0).

## Технические задачи

### 5.1 Explainability-слой

- Модель: `Explanation { fact_id, title, what_it_means, how_measured { source_api, method, sample_window, error_margin }, action_consequences[], undo_path, references[] }` — типизированная структура, привязанная к каждому UI-значению, а не tooltip-строки россыпью.
- Контент — отдельные ресурсные файлы (JSON/markdown), review'ится как продуктовый текст: без запугивания, без преувеличений, «можно ничего не делать» — валидная рекомендация.
- UI: единый паттерн (иконка → side-pane), «копировать объяснение» для суппорта/форумов.
- Аудит всех экранов Phase 1–4 на покрытие explanation'ами — чек-лист, DoD-пункт.

### 5.2 UX-полировка

- Onboarding: первый запуск объясняет философию (нет плацебо, всё обратимо), запрашивает согласия (фоновые снапшоты, крэш-репорты) по одному и honestly.
- Состояния: пустые (нет истории трендов — что сделать), деградированные (нет admin, WSUS, отключён System Restore), ошибки collector'ов (partial errors из Phase 0 — наконец показать по-человечески).
- Perf: холодный старт UI < 2 с до интерактивности (отложенная загрузка данных), отсутствие UI-фризов > 100 мс (профилирование WPA).
- Доступность: полная клавиатурная навигация, UIA-имена для Narrator, контраст WCAG AA, high-contrast темы.
- Локализация: вынести все строки (включая explanation-контент) в ресурсы; ship en+ru; псевдолокализация-тест на непереведённые строки.
- UWP StartupTask-управление (отложенное из Phase 1) — добавить.

### 5.3 Автообновление

- Схема: канал-манифест (JSON) на HTTPS-домене продукта: version, url, SHA-256, минимально-поддерживаемая версия, staged-процент. Клиент проверяет раз в сутки фоновой задачей.
- Скачивание → проверка SHA-256 → **проверка Authenticode-подписи и издателя** (`WinVerifyTrust` + сверка thumbprint'а сертификата с вшитым) → установка при следующем запуске или по кнопке. Никогда — молчаливый рестарт приложения посреди работы.
- Staged rollout: манифест содержит процент; клиент детерминированно (hash от machine_fingerprint) решает, попал ли в волну — откат волны = правка манифеста.
- Rollback: предыдущий инсталлятор хранится локально; «вернуть предыдущую версию» в настройках. Миграции БД (Phase 0-механизм) должны быть forward-only совместимы: старая версия приложения с новой БД — отказывается стартовать с внятным сообщением, не портит данные.
- Безопасность: манифест — только HTTPS с pinning на свой домен; подпись манифеста (Ed25519, ключ офлайн) поверх TLS. Updater — главный вектор атаки на пользователей, проектировать с threat model-документом.

### 5.4 Подпись и дистрибуция

- Сертификат: OV code signing (EV ускоряет SmartScreen-репутацию — оценить бюджет; для системной утилиты EV рекомендован). Подпись **всех** бинарей, включая elevated helper и CLI; timestamping (RFC 3161) обязателен.
- SmartScreen: репутация набирается количеством установок — заложить «тихий» период после первого публичного релиза; сабмит в Microsoft для ускорения.
- Антивирусы: предрелизный прогон каждой RC на VirusTotal; whitelisting-сабмиты (Defender, Kaspersky, Avast, ESET) — процесс, заведённый ещё в Phase 1, здесь формализуется в чек-лист релиза.
- Инсталлятор: финализировать MSIX-vs-MSI решение Phase 1 с учётом автообновления (unpackaged+MSI со своим updater'ом — более предсказуемо для этого класса ПО; MSIX упростил бы, но full-trust+elevated helper в packaged-модели надо перепроверить на реальных сборках).

### 5.5 Крэш-репорты и диагностика

- `SetUnhandledExceptionFilter` + minidump (`MiniDumpWriteDump`, отдельный процесс-writer, чтобы дамп писался не из упавшего процесса). Локально всегда; отправка — только opt-in, с показом, что именно уходит.
- Диагностический экспорт для суппорта: zip с логами, снапшотом настроек, последними ошибками collector'ов — по кнопке, пользователь видит содержимое.

## Зависимости от предыдущих фаз

Все (полировка поверх Phase 1–4). Механизм миграций БД — Phase 0; фоновые задачи — Phase 2; elevated helper — Phase 1.

## Риски и митигация

| Риск | Митигация |
|------|-----------|
| Updater скомпрометирован/MITM | Подпись манифеста офлайн-ключом + Authenticode + pinning; threat model review; ключи в HSM/офлайн |
| SmartScreen пугает первых пользователей | EV-сертификат (если бюджет), ранний сабмит, честная страница «почему предупреждение» |
| Explainability-контент разбухает и устаревает | Контент в ресурсах с owner'ом; fact_id-линковка — CI-проверка «каждый fact имеет explanation» |
| Миграция БД у реальных пользователей падает | Автобэкап БД перед миграцией; телеметрия миграций в первых волнах staged rollout |
| WinUI 3-доступность имеет пробелы | Ранний Narrator-прогон; известные пробелы — документировать, критические — кастомными UIA-пирами |

## Критерии готовности (Definition of Done)

- [ ] 100% значений/статусов/действий имеют Explanation (CI-проверка + ручной аудит).
- [ ] Коридорные тесты: ≥ 80% успеха трёх ключевых сценариев на людях «не-айтишниках».
- [ ] Автообновление: цепочка v(N-1)→vN на матрице ОС, включая повреждённый пакет (отказ с сохранением старой версии), подмену манифеста (отказ), rollback.
- [ ] Все бинари подписаны, timestamp'ованы; VirusTotal RC — детекты разобраны/отсабмичены.
- [ ] Полная клавиатурная навигация + Narrator-прогон ключевых экранов.
- [ ] Крэш → дамп пишется, следующий запуск предлагает отправку; экспорт диагностики работает.
- [ ] en+ru полные; псевдолокализация не находит hardcoded-строк.
- [ ] Чек-лист релиза 1.0 (подпись, манифест, страница загрузки, канал отката) прогнан на staging.

## Тестирование

- **Юзабилити:** 5–8 сессий коридорных тестов на реальных пользователях, по протоколу, с фиксацией провалов сценариев.
- **Updater:** авто-тесты цепочек обновления на VM (старые версии из архива CI); негативные — битые хэши, отозванная подпись, downgrade-атака (манифест со старой версией — клиент отказывается).
- **Доступность:** Accessibility Insights for Windows, ручной Narrator-прогон.
- **Perf:** WPA-трейсы холодного старта на HDD-машине (worst case).
- **Регресс:** полный ручной регресс Phase 1–4 функциональности на матрице ОС + автоматизированный smoke.

## Оценка трудозатрат

**4–5 недель** разработки + **1–2 недели** релизного цикла (RC, сабмиты, репутация). Explainability-инфраструктура ~1 нед, контент ~1 нед (параллелится с остальным); UX-полировка ~1.5 нед; updater ~1.5 нед; подпись/дистрибуция/чек-листы ~0.5 нед; коридорные тесты и правки ~1 нед.

## Открытые вопросы

- EV vs OV сертификат — бюджетное решение до начала фазы.
- Хостинг манифеста/пакетов (свой домен + CDN) — выбрать; требование: HTTPS, стабильность, логи для staged-метрик.
- Монетизация 1.0 (free/freemium) — продуктовое решение, влияет на EULA/privacy тексты этой фазы.
