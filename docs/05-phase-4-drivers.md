# Phase 4: Обновление драйверов — официальные источники, HWID-матчинг, restore points

## Цель фазы

Модуль драйверов, который:

- инвентаризует устройства и текущие драйверы (Hardware ID, версия, дата, подпись, INF);
- находит обновления **только** в официальных источниках: Windows Update (через WU Agent API), Microsoft Update Catalog, официальные каналы Intel/NVIDIA/AMD;
- матчит строго по Hardware ID с корректным ranking'ом (никаких «универсальных баз»);
- перед любой установкой создаёт restore point + бэкап текущего драйвера; после — верифицирует, что устройство поднялось;
- честно показывает «драйвер актуален» — это нормальный и частый ответ.

Измеримо: на тестовой матрице устройств предложения совпадают с тем, что предлагает Windows Update / вендорские утилиты; ни одной установки без restore point + driver backup; откат работает.

## Границы (Scope)

**Входит:** инвентаризация PnP; поиск через WUA API (категория Drivers); прямой поиск в Microsoft Update Catalog (fallback/расширение); GPU-каналы NVIDIA/AMD/Intel (метаданные версий; скачивание — с официальных доменов вендора); backup/restore драйвера; restore point; верификация после установки.

**НЕ входит:**
- Любые сторонние базы драйверов и «драйвер-паки».
- BIOS/UEFI/firmware-апдейты (отдельный класс рисков — backlog с отдельным решением).
- Массовая «обновить всё» кнопка в v1 — только по-устройственно (снижает blast radius).
- Beta/preview драйверы вендоров.

## Технические задачи

### 4.1 Инвентаризация устройств

- SetupAPI/CfgMgr32: `SetupDiGetClassDevsW(DIGCF_ALLCLASSES|DIGCF_PRESENT)` → `SetupDiGetDeviceRegistryPropertyW` / `CM_Get_DevNode_PropertyW`: `HardwareID` (список, по убыванию специфичности), `CompatibleIDs`, класс, производитель.
- Текущий драйвер: `SetupDiEnumDriverInfoW`/`SetupDiGetDriverInfoDetailW` либо device properties `DEVPKEY_Device_DriverVersion`, `DriverDate`, `DriverInfPath`, `DriverProvider`. Подпись INF — `DEVPKEY_Device_DriverInfSection` + каталог.
- Проблемные устройства: `CM_Get_DevNode_Status` → code 28 (нет драйвера), 43 и т.д. — отдельный список «требуют внимания», это самая полезная категория для пользователя.
- Всё — очередной collector (`drivers.inventory`) в контракте Phase 0, попадает в снапшоты/тренды.

### 4.2 Поиск через Windows Update Agent API (основной путь)

- COM `IUpdateSession`/`IUpdateSearcher` (`wuapi.h`): критерий `"IsInstalled=0 and Type='Driver'"`; `IUpdateSearcher::put_ServerSelection(ssWindowsUpdate)` — искать на WU, даже если в системе WSUS.
- Результат `IUpdateDriver`-интерфейсы: `DriverHardwareID`, `DriverVerDate`, `DriverProvider`, `DriverClass` — сопоставление с инвентаризацией напрямую.
- Установка: `IUpdateDownloader` + `IUpdateInstaller` — этим путём инсталляция идёт штатным механизмом Windows (подпись/доверие проверяет ОС). Требует admin → elevated helper.
- Ограничение: WU выдаёт только то, что вендор пометил как auto-offered; более новые «optional» драйверы ищутся через Catalog (4.3).

### 4.3 Microsoft Update Catalog (расширенный поиск)

- **Публичного REST API нет** — это ключевой риск фазы. Два легальных пути:
  1. WUA API с `ssWindowsUpdate` + флаг `BrowseOnly` updates (`IUpdateSearcher::Search("IsInstalled=0 and Type='Driver' and BrowseOnly=1")`) — покрывает большую часть optional-драйверов без скрейпинга; **рекомендация: только этот путь в v1**.
  2. HTML-скрейпинг catalog.update.microsoft.com — хрупко и в серой зоне; отклонить, зафиксировать ADR'ом.
- Скачанные пакеты (.cab) проверять: `WinVerifyTrust` (Authenticode), издатель Microsoft/вендор; распаковка и установка через `DiInstallDriverW` (newdev.h) или `pnputil /add-driver /install` (elevated helper).

### 4.4 Вендорские каналы GPU

- NVIDIA: публичный JSON-endpoint службы GeForce-обновлений для матчинга по PCI DevID; скачивание — только с download.nvidia.com. Проверить актуальные условия использования до имплементации.
- AMD: страницы auto-detect и JSON-каналы Adrenalin; скачивание с официального CDN AMD.
- Intel: Intel Driver & Support Assistant не имеет публичного API — для Intel ограничиться WU/Catalog-путём в v1.
- Общий принцип: вендорский канал даёт **метаданные** «есть версия X»; ссылка на скачивание обязана вести на домен вендора; хэш/подпись проверяются перед установкой (`WinVerifyTrust` + издатель = вендор).
- Юридически: перепроверить EULA каждого канала; если авто-скачивание запрещено — деградация до «открыть страницу загрузки вендора» (это тоже честная фича).

### 4.5 Restore point + driver backup

- Restore point: `SRSetRestorePointW` (`srclient.dll`), `BEGIN_SYSTEM_CHANGE`/`END_SYSTEM_CHANGE`. Подводный камень: Windows молча пропускает создание, если точка создавалась < 24 ч назад (`SystemRestorePointCreationFrequency` в реестре) — **проверять результат и sequence number**, при пропуске сообщать пользователю, что откат будет на существующую точку (показать её дату). Если System Restore отключён — предупредить и предложить включить; установка без какого-либо отката запрещена политикой продукта.
- Driver backup: экспорт текущего пакета `pnputil /export-driver <oem#.inf> <dir>` (или `DriverStore Explorer`-подход через SetupAPI) в `%ProgramData%\Tracewell\DriverBackups\<device>\<version>` до установки. Откат = `pnputil /add-driver` бэкапа + `DiInstallDevice`/rollback через Device Manager API (`DIF_...`) либо штатный `DEVPKEY`-based rollback.
- Верификация после установки: rescan (`CM_Reenumerate_DevNode`), проверка problem code == 0 и версии; при деградации — авто-предложение отката.

### 4.6 UX-контур

- По каждому устройству: текущий драйвер / доступный / источник / что изменится / «зачем обновлять» (и явное «можно не обновлять, если всё работает» — анти-паттерн конкурентов).
- Прогресс установки с журналом; reboot-required — честно, без форс-ребута.

## Зависимости от предыдущих фаз

- Phase 0: collectors/снапшоты (инвентарь в трендах), журнал.
- Phase 1: elevated helper (все установки), ActionJournal, UI.
- Phase 2: hardware inventory pathways (enumerate дисков/шин переиспользуется), опыт elevated-операций.

## Риски и митигация

| Риск | Митигация |
|------|-----------|
| Установка драйвера ломает систему (главный) | Restore point (проверенный!) + driver backup + верификация + авто-rollback-предложение; на VM — прогон worst-case (GPU-драйвер, сторедж-драйвер запрещён к обновлению в v1 стоп-листом) |
| Нет публичного Catalog API | BrowseOnly-поиск через WUA как основной; скрейпинг отклонён ADR'ом; готовность к тому, что покрытие меньше, чем у «драйвер-бустеров» — компенсируется честностью |
| Вендорские каналы меняются/закрываются | Фича-флаги на каждый канал (отключение без релиза — конфиг с подписью); деградация до «открыть сайт вендора» |
| HWID-матчинг: ranking (specific vs compatible ID) | Реализовать порядок из документации INF ranking; никогда не предлагать матч только по CompatibleID без явной пометки «совместимый, не точный» |
| WSUS/корпоративные политики блокируют WU-поиск | Детект `UseWUServer` политики → сообщение «управляется организацией», модуль в read-only |
| Storage/chipset драйвер — окирпичивание | Стоп-лист классов в v1: storage controllers, chipset — read-only показ версии, без кнопки установки |

## Критерии готовности (Definition of Done)

- [ ] Инвентарь устройств полон (сверка с Device Manager) на всей матрице железа.
- [ ] Предложения по драйверам: нет ни одного случая предложения драйвера с несовпадающим specific HWID; compatible-матчи помечены.
- [ ] Полный цикл на VM и живой машине: поиск → restore point (проверен sequence) → backup → установка → верификация → откат — все шаги журналируются и работают.
- [ ] Сценарий «restore point пропущен из-за 24h-правила» корректно обнаружен и показан.
- [ ] «Все драйверы актуальны» — частый и нормально оформленный результат (не «0 проблем найдено», а конкретика: N устройств проверено, источники).
- [ ] WSUS-машина: модуль честно деградирует.
- [ ] Стоп-лист storage/chipset действует.

## Тестирование

- **Unit:** HWID-ranking (фикстуры реальных HWID-списков), парсинг версий/дат, логика деградации каналов.
- **Интеграционные (Hyper-V, чекпоинты обязательны):** установка/откат на виртуальных устройствах; намеренная установка старого драйвера → модуль предлагает обновление → полный цикл.
- **Живое железо:** по одной машине NVIDIA/AMD/Intel GPU; ноутбук с OEM-драйверами (частый конфликт «vendor generic vs OEM-custom» — проверить, что не предлагаем generic поверх OEM без пометки).
- **Негативные:** отключённый System Restore, WSUS-политика, оффлайн-режим, повреждённый скачанный .cab (подпись не проходит → отказ установки).

## Оценка трудозатрат

**6–8 недель.** Инвентаризация ~1 нед; WUA-поиск+установка ~1.5 нед; BrowseOnly/Catalog-путь ~1 нед; вендорские каналы (2 из 3) ~1–1.5 нед; restore point+backup+rollback ~1 нед; UX ~1 нед; тесты на матрице ~1 нед (частично перекрывается).

## Открытые вопросы

- Юридическая проверка условий NVIDIA/AMD-каналов — **до начала фазы** (блокер для 4.4).
- Обновлять ли драйверы устройств, отсутствующих сейчас (DIGCF_PRESENT off) — предложение: нет.
- Политика для ARM64-устройств — отложить до появления ARM64-сборки (Phase 5+).
